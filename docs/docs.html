§00 · OVERVIEW

AOS Manual

AOS (Agent Operations Stack) is the operations layer for AI coding agents. It gives every project a portable memory, enforceable guardrails, an automatic audit trail, real verification, and a local console — stored entirely as plain files under ~/.aos that you own and any agent can read.

It is three thin parts:

  1. The Spec — a file convention: context packs, policies, playbooks, run records, audit logs.
  2. Skills + Hooks — Claude Code integration that loads context, gates commands, and audits actions automatically, plus a six-stage ticket pipeline.
  3. The Console — a read-only local dashboard: decision queue, run states, verification verdicts, token economics.

AOS is not an orchestration framework, not a platform, and not an SDK you embed. Your runtime (Claude Code today; anything file-capable tomorrow) does the work — AOS records, gates, briefs, and reports.

§01 · INSTALLATION

Installation

REQUIREMENTS

  • macOS or Linux
  • git
  • Node.js ≥ 18 (node -v)
  • Claude Code, for the skills/hooks integration (the CLI and console work without it)

ONE-LINE INSTALL

$ curl -fsSL https://raw.githubusercontent.com/albsugy/aos/main/install.sh | bash

The installer: checks prerequisites → clones the app to ~/.local/share/aos → installs dependencies → links ~/.local/bin/aos → adds that directory to your PATH (zsh, bash, or fish) if it isn't already → verifies with aos version. Re-running it updates in place.

OVERRIDES

VariableDefaultPurpose
AOS_INSTALL_DIR~/.local/share/aosWhere the app lives
AOS_BIN_DIR~/.local/binWhere the aos symlink goes
AOS_REPO_URLthe GitHub repoInstall from a fork or mirror
AOS_HOME~/.aosWhere all data lives (runtime, not install-time)

MANUAL INSTALL

$ git clone https://github.com/albsugy/aos.git && cd aos
$ npm install --omit=dev
$ ln -sf "$PWD/bin/aos.js" ~/.local/bin/aos
§02 · QUICKSTART

Quickstart

From zero to your first gated, audited, verified run:

# 1 — register a repo as an AOS project
$ cd your-repo
$ aos init
✔ Registered project "your-repo"
✔ Spec scaffolded at ~/.aos/projects/your-repo
✔ Skills installed to .claude/skills/
✔ Hooks wired in .claude/settings.json

2 — Fill in the two files that matter (both under ~/.aos/projects/<id>/):

  • context/pack.md — what every agent must know: architecture, conventions, boundaries, gotchas.
  • policy.yaml — what agents may do: forbidden and gated commands, verification contracts. See §04.

3 — Start a new Claude Code session in the repo (hooks load at session start) and run a ticket:

/aos-ticket PROJ-123: add rate limiting to the public API

The pipeline runs intake → plan → implement → verify → package → learn. Gated commands (like git push) pause for your approval. When it finishes, the run is awaiting-review with a PR draft in its outcome.md.

$ aos status        # every project, run, and what needs you
$ aos console       # dashboard at http://127.0.0.1:4560
WORKS WITH ZERO WORKFLOW CHANGE Even if you never run /aos-ticket, the hooks alone give you session context injection, policy gates, an audit trail, and token accounting on every ordinary Claude Code session.
§03 · CONCEPTS

Concepts

PROJECT

A registered repo (or several repos) mapped to a memory home under ~/.aos/projects/<id>/. The mapping lives in ~/.aos/registry.yaml; AOS resolves which project you're in from your working directory (longest path match, symlinks resolved).

RUN

One unit of work — usually one ticket. A run owns a folder of records: ticket.md, plan.md, verification.md, outcome.md, audit.jsonl, meta.json. One run per project is active at a time; hook audit lines flow into it. Audit outside any run still lands in the project-level audit.jsonl.

RUN STATES

StateMeaning
in-progressAgent is working (set by run start)
blockedAgent stopped — needs your input to continue
awaiting-reviewVerified and packaged — waiting for your review (default of run finish)
doneYou reviewed it
shippedMerged/deployed

LEVERAGE RATIO

The percentage of finished runs whose verification passed on the first attempt. This is the number that tells you how much agent output ships untouched, safely — the whole point of operating well. It's shown per project in aos status and the console.

GATES

Policy rules matched against every Bash command an agent tries to run: forbidden → blocked outright · gated → held for your approval · everything else → flows (normal Claude Code permissions still apply on top).

§04 · DIRECTORY LAYOUT

Directory layout

~/.aos/
├── registry.yaml                # project id → repo paths
└── projects/<id>/
    ├── context/
    │   ├── pack.md              # the brief — loaded into every session
    │   └── decisions.md         # append-only decision log (last 40 lines load)
    ├── policy.yaml              # gates + verification contracts
    ├── learnings.md             # compounding gotchas (last 30 lines load)
    ├── playbooks/               # extracted repeatable procedures
    ├── state.json               # active run pointer
    ├── audit.jsonl              # actions outside any run
    ├── sessions.jsonl           # token usage per session
    └── runs/<YYYY-MM-DD>-<ticket>/
        ├── meta.json            # state, verification, attempts, tokens
        ├── ticket.md            # source + acceptance criteria
        ├── plan.md
        ├── audit.jsonl          # every action during this run
        ├── verification.md      # contract results + adversarial findings
        └── outcome.md           # summary, decisions, risks, PR draft

Everything is plain text. git init ~/.aos gives your entire operation version history; grep works; nothing needs AOS to be readable in ten years.

§05 · POLICY REFERENCE

policy.yaml reference

version: 1
plan_gate: auto            # auto | ask — must a human approve plan.md first?

tiers:
  forbidden:               # blocked outright, agent is told why
    - pattern: 'push\s+[^|;&]*(--force|-f)\b'
      reason: Force-push is forbidden by policy
  gated:                   # held for human approval
    - pattern: '\bgit\s+push\b'
      action: git-push
    - pattern: '\bdeploy\b'
      action: deploy

verification:
  adversarial_review: true # require a skeptic subagent in /aos-verify & /aos-ticket
  contracts:               # deterministic checks run by `aos verify`
    - name: tests
      command: npm test
      required: true       # a required failure = verdict FAIL (exit 1)
    - name: lint
      command: npm run lint
      required: false
      timeout_ms: 120000   # optional, default 10 minutes

SEMANTICS

  • pattern is a JavaScript regex, matched case-insensitively against the full Bash command string.
  • Order of evaluation: forbidden → gated → allow. First match wins within a tier.
  • A broken regex is skipped silently — a bad pattern can never take the gate down.
  • Anything not matched is allowed by AOS; Claude Code's own permission system still applies.
  • Contract commands run with your shell in the repo directory — same trust model as package.json scripts.
  • Missing or invalid policy.yaml → safe defaults (force-push and rm -rf / forbidden; push/merge/deploy gated).
DESIGN INTENT Gates express blast radius, not distrust. Reads, tests, and edits flow freely; the moment a command can affect the world outside the working tree — push, merge, deploy, send — a human enters the loop.
§06 · HOOKS

Hooks

aos init wires four Claude Code hooks into the repo's .claude/settings.json. They are the enforcement layer — active in every session, no skill invocation needed.

HookFiresWhat it does
SessionStartnew sessionInjects the context pack, recent decisions, learnings, and open runs into the agent's context
PreToolUseevery Bash callEvaluates the command against policy.yaml: forbidden → denied with reason; gated → surfaces as a Claude Code approval prompt tagged [aos policy]
PostToolUseevery tool callAppends a summarized audit line to the active run (or project) audit.jsonl
SessionEndsession closeParses the transcript for token usage; records per session and per active run

LAUNCHER RESOLUTION

Hook commands call the stable launcher ("$HOME/.local/bin/aos", with a PATH fallback) rather than a pinned install path — so aos update, reinstalls, and switching between a dev checkout and the installed package never break existing repos. Re-running aos init migrates hooks written by older versions.

FAILURE POLICY

Hooks are wrapped so that any internal error exits silently with code 0 — a broken AOS install can never break your Claude Code session (the command ends in || true by design). The trade-off: if AOS misbehaves, check troubleshooting rather than expecting loud errors.

NON-AOS REPOS

Hooks resolve the project from the session's working directory. In directories that aren't registered, all hooks no-op instantly — installing AOS globally has zero effect on unregistered repos.

§07 · SKILLS

Skills

/aos-ticket <ticket>

The full pipeline. Six stages, none skippable:

  1. Intake — starts a run, records the ticket and explicit acceptance criteria (asks you if ambiguous)
  2. Plan — writes plan.md; pauses for approval if plan_gate: ask
  3. Implement — normal agent work on a branch, gated and audited by the hooks
  4. Verify — runs aos verify contracts, then spawns an adversarial reviewer instructed to refute the work against the acceptance criteria; loops until clean or blocked
  5. Package — writes outcome.md: summary, changes, decisions, risks, test steps, PR draft; finishes the run as awaiting-review
  6. Learn — appends learnings and decisions to project memory; proposes a playbook when a pattern repeats

/aos-verify

Stage 4 standalone — verify anything, anytime: contracts + skeptic subagent, findings recorded to the active run's verification.md when one exists.

/aos-learn

Distills the current session into learnings.md, decisions.md, pack updates, and playbook candidates. Use at the end of significant unstructured work or when you say "remember this."

/aos-ask <question>

Answers "why did we…", "what happened with…" questions from run history, with file:line citations. Says so plainly when history doesn't contain the answer.

§08 · CLI REFERENCE

CLI commands

CommandDescription
aos init [--name <n>]Register the current repo as a project: scaffolds the spec under ~/.aos, copies skills into .claude/skills/, wires hooks into .claude/settings.json. Idempotent — safe to re-run.
aos statusAll projects: run counts by state, active run, leverage ratio, token totals, and anything awaiting review.
aos context [--project <id>]Print exactly what agents get injected at session start.
aos run start --ticket <id> [--title <t>]Create a run folder, scaffold ticket.md, set it active. Prints the folder and plan_gate.
aos run finish [--state <s>]Finish the active run (default awaiting-review) and clear the active pointer.
aos run state <state>Set the active run's state: in-progress · blocked · awaiting-review · done · shipped.
aos run listAll runs for the current project with state and verification.
aos verifyRun the policy's verification contracts here. Writes verification.md on the active run. Exit 0 = all required passed; exit 1 otherwise.
aos find <query>Search all project memory (runs, decisions, learnings, audit) — prints file:line matches.
aos console [--port <p>]Serve the local console (default 127.0.0.1:4560).
aos projectsList registered projects and their memory homes.
aos versionPrint the installed version.
aos updateUpdate in place: git pull --ff-only + reinstall dependencies.
aos hook <name>Internal — the entry points Claude Code hooks call (pre-tool, post-tool, session-start, session-end).

Most commands resolve the project from your working directory; pass --project <id> to override.

§09 · THE CONSOLE

The console

aos console serves a read-only dashboard at http://127.0.0.1:4560 (change with --port). It refreshes every 5 seconds from the files on disk — no database, no daemon.

WHAT YOU SEE

  • Header KPIs — projects, runs that need you, total tokens.
  • Decision queue — every run that is blocked or awaiting-review, across all projects. This is the batch-review surface: clear it twice a day.
  • Project tables — runs with state and verification chips, attempts, tokens, recency. Finished rows recede visually; live work stands forward.
  • Run drawer — click any run: outcome, verification report, the full audit timeline (gate decisions in amber, verification verdicts in green/red), ticket, and plan.
  • Leverage ratio — per project, the % of finished runs that verified clean on the first attempt.
LOCAL ONLY The server binds to 127.0.0.1 — it is not reachable from your network. There is no authentication because there is no exposure. Remote/mobile access arrives with the v0.2 Telegram approvals and the v0.5 sync server.
§10 · DATA & PRIVACY

Data & privacy

  • Everything is local. AOS makes no network calls. No telemetry, no accounts, no cloud.
  • What is stored: your context packs and policies (authored by you), run records and audit summaries (tool names, file paths, truncated commands), and token counts parsed from session transcripts.
  • What is not stored: full transcripts, file contents, or credentials. Audit lines are summaries, capped in length.
  • Employer code: run folders describe work done in a repo (tickets, plans, outcomes). If that's sensitive, point AOS_HOME at an approved volume, or keep per-employer data separate by setting AOS_HOME per machine/shell profile.
  • Back it up like it matters — this data is your compounding asset: git init ~/.aos && git -C ~/.aos add -A && git -C ~/.aos commit -m snapshot
§11 · TEAMS & ORGANIZATIONS

Teams & organizations PLANNED — v0.5+

The solo stack is free forever. The paid tier extends the same files across people:

  • Team workspaces — shared context packs, policies, and playbooks; every teammate's agents brief from one source of truth, and one teammate's learning compounds for everyone.
  • Self-hostable sync server — project state, decision queues, and audit shared across machines; the file spec stays the source of truth, sync is transport.
  • Approval routing — gates routed by role: a junior's deploy goes to a senior; a client-facing send goes to the account owner.
  • Org-wide audit & policy baselines — one exportable trail of every agent action in the company, and guardrail floors no project can drop below. Built with regulatory oversight requirements (EU AI Act, Aug 2026) in mind.
  • Cross-organization sharing — agencies and their clients on the same trail: the client sees, approves, and trusts what agents do in their business.

Interested in the team tier? Open an issue on GitHub — early design partners shape the spec.

§12 · TROUBLESHOOTING

Troubleshooting

`aos: command not found`

Your shell hasn't picked up the PATH change. Restart the shell, or run export PATH="$HOME/.local/bin:$PATH". Verify with which aos.

HOOKS DON'T SEEM TO FIRE

  • Hooks load at session start — start a new Claude Code session after aos init.
  • Check .claude/settings.json in the repo contains the four aos hook entries.
  • Confirm the repo is registered: aos projects should list its path. If you moved the repo, re-run aos init.

GATE DIDN'T TRIGGER ON A COMMAND

  • Test the pattern directly: does the project resolve? cd into the repo and run aos context — an error means the directory isn't matched to a project.
  • Patterns match the whole Bash command string, case-insensitively. Compound commands (a && b) are matched as one string.
  • Remember gates apply to Bash tool calls from the agent — not to commands you type in your own terminal.

`node: bad option` OR SYNTAX ERRORS

Node < 18. Check node -v; upgrade and re-run the installer.

CONSOLE PORT IN USE

aos console --port 4561

MOVED OR RENAMED A REPO

Re-run aos init in the new location — it appends the new path to the same project. Stale paths in ~/.aos/registry.yaml can be deleted by hand; it's YAML, edit freely.

§13 · UPDATE & UNINSTALL

Update & uninstall

UPDATE

$ aos update
# or re-run the install one-liner — same effect

UNINSTALL

# remove the app + launcher
$ rm -rf ~/.local/share/aos ~/.local/bin/aos

# per repo: remove skills + the aos hook entries
$ rm -rf .claude/skills/aos-*        # then delete aos hooks from .claude/settings.json

# your data — keep it. But if you must:
$ rm -rf ~/.aos

Nothing else is touched: no launch agents, no system files, no shell functions — one PATH line in your shell rc is the only trace, marked # added by aos installer.
